Part 1

Foundations & Core API

Setting the stage and introducing the Container concept

The Problem

Why new Thread() is Dangerous

In standard Java SE, creating a thread is as simple as new Thread(task).start(). However, in an Enterprise Server environment, this causes chaos.

Unmanaged Execution

  • The server does not know the thread exists.
  • No Security: The logged-in user's identity is lost.
  • No Transactions: Database connections can leak or fail to commit.
  • No Limits: Can easily exhaust server memory and CPU under high load.
💀

Chaos: Unmanaged Thread

  • No security identity
  • No JNDI or CDI context
  • Rogue execution
The Solution

The "Container" Concept

🏛️

Order: Managed Thread

  • Container-supervised
  • Context explicitly propagated
  • Pooled & Monitored

Jakarta Concurrency

The application server must control all resources to guarantee stability and security. Jakarta Concurrency formalizes thread management by providing Managed versions of standard Java concurrent utilities.

Instead of creating threads manually, the server maintains optimal Thread Pools and loans threads to your application when needed.

Core API

The Workhorse: ManagedExecutorService

The direct substitute for Java's traditional ExecutorService. Submit Runnable or Callable tasks to the container safely.

TaskSubmissionDemo.java 
@Stateless
public class ReportService {

  // 1. Injection via @Resource
  @Resource
  ManagedExecutorService executor;

  public void generateHeavyReport() {
    // 2. Submit a Runnable
    executor.submit(() -> {
      // Runs in a managed thread!
      System.out.println("Building PDF...");
    });
  }
}

How it Works

  • You inject the service using @Resource.
  • You submit your background task (e.g. sending an email, generating a file).
  • The server pulls an available thread from its configured pool, attaches your application's context to it, runs the task, and returns the thread to the pool.

Behind the Code

HubService.java — The Actual Backend

This is the real Java code running on the server when you click "Submit Task" in the demo.

HubService.java 
@ApplicationScoped
public class HubService {

    @Resource
    private ManagedExecutorService executor;

    public TaskResult processTask(String taskName) {
        Future<TaskResult> future = executor.submit(() -> {
            Thread.sleep(500); // Simulate work
            return new TaskResult(
                taskName,
                Thread.currentThread().getName(),
                "COMPLETED"
            );
        });
        return future.get(); // Blocks until managed thread finishes
    }
}
💡 The container injects a real ManagedExecutorService. The submit() call returns a Future proving the task ran on a managed thread, not the HTTP thread.
Part 2

Timing & Safety

Scheduling jobs and securing context propagation

Scheduling

The Scheduler: ManagedScheduledExecutorService

Replaces old java.util.Timer. Perfect for periodic cleanup or deferred actions.

Schedule Patterns

  • schedule(): Delay execution once.
  • scheduleAtFixedRate(): Exact periodicity (e.g. every 5 seconds).
  • scheduleWithFixedDelay(): Delay between the end of one run and start of next.
CleanupTask.java 
@Singleton
@Startup
public class CleanupTask {

  @Resource
  ManagedScheduledExecutorService scheduler;

  @PostConstruct
  public void init() {
    // Runs exactly 5 seconds after deployment!
    scheduler.schedule(() -> {
      log.info("Running initial system cleanup...");
    }, 5, TimeUnit.SECONDS);
  }
}

Behind the Code

TickerService.java — The Actual Backend

This service uses @PostConstruct to start a scheduled task the moment the server deploys it.

TickerService.java 
@ApplicationScoped
public class TickerService {

    @Resource
    private ManagedScheduledExecutorService scheduler;

    private List<StockPrice> latestPrices = new CopyOnWriteArrayList<>();

    @PostConstruct
    public void startTicker() {
        scheduler.scheduleAtFixedRate(() -> {
            for (StockPrice stock : latestPrices) {
                double variation = (Math.random() - 0.5) * 5;
                stock.price += variation;
            }
        }, 0, 3, TimeUnit.SECONDS);
    }
}
💡 scheduleAtFixedRate(task, 0, 3, SECONDS) means: start immediately, then repeat every 3 seconds. The container manages the scheduling thread automatically.

ContextDemoService.java — The Actual Backend

This service uses ContextService and ManagedThreadFactory — the two remaining Jakarta Concurrency APIs.

ContextDemoService.java 
@ApplicationScoped
public class ContextDemoService {

    @Resource
    private ContextService contextService;

    @Resource
    private ManagedThreadFactory threadFactory;

    public TaskResult runWithContext(String user, String role) {
        Callable<String> task = () -> "User: " + user;

        // Wrap the task with the current context snapshot
        Callable<String> contextualTask =
            contextService.contextualCallable(task);

        return new TaskResult("ContextProxy",
            "Proxied Object Created", "SUCCESS");
    }
}
💡 contextService.contextualCallable(task) wraps the task with a snapshot of the caller's security identity, JNDI, and classloader — so even when it runs later on a different thread, the context is preserved.

Practical Use Cases

Applying the utilities to real-world app scenarios

Use Case A

Asynchronous Notifications

The "Fire and Forget" Pattern

Scenario: A new user registers for your system. Sending a Welcome Email synchronously over SMTP takes 2-3 seconds, locking up the user's browser completely.

Solution: By offloading the email task to the ManagedExecutorService, the user gets an instant response while the email server processes the background request securely.

RegistrationService.java 
@Stateless
public class RegistrationService {

  @Resource ManagedExecutorService executor;

  public void registerNewUser(User user) {
      db.save(user);  // Sync database commit

      // Fire & Forget Response
      executor.submit(() -> {
          emailService.sendWelcomeEmail(user);
      });
  }
}

Behind the Code

PizzaService.java — The Actual Backend

This service forks 3 parallel async tasks using executor.invokeAll() and waits for all of them to complete.

PizzaService.java 
@ApplicationScoped
public class PizzaService {

    @Resource
    private ManagedExecutorService executor;

    public List<TaskResult> orderPizza(String name) {
        Callable<TaskResult> emailTask = () -> {
            Thread.sleep(400);
            return new TaskResult("Email",
                Thread.currentThread().getName(), "DONE");
        };
        Callable<TaskResult> pushTask  = () -> { /* ... */ };
        Callable<TaskResult> statsTask = () -> { /* ... */ };

        // Run ALL tasks in parallel on managed threads
        List<Future<TaskResult>> futures =
            executor.invokeAll(List.of(
                emailTask, pushTask, statsTask));

        for (Future<TaskResult> f : futures)
            results.add(f.get());
        return results;
    }
}
💡 invokeAll() submits all tasks simultaneously to the managed pool. Each runs on its own thread — the server handles scheduling, context, and cleanup for all of them.

Why Jakarta Concurrency is Essential

It bridges the gap between Java SE's raw threading power and the Enterprise container's need for strict resource governance.

  • Safe: Threads don't leak or crash the server.
  • Secure: Identity is propagated naturally.
  • Performant: Pooling decreases thread setup costs.
